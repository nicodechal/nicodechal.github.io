###FP-Growth算法介绍

FP-growth (Frequent-Pattern Growth)是数据挖掘中用于挖掘频繁项集的经典算法之一。相较于 Apriori 算法，该算法消除了候选项集，并减少了对数据库扫描的次数，因而效率更高。

具体算法思路可以参考数据挖掘教材 data mining concepts and techniques 第六章的内容。

###Spark中的FP-Growth算法

Spark 中 FP-Growth算法 实现了分布式环境下的 FP-Growth 算法。

#####向树中增加事务

增加事务主要通过 add 方法实现，过程可以概括为，对事务中的元素进行遍历，对于每个元素($item$)做两件事情：1. 更新 Summaries，2. 更新结点树。

更新 Summaries

对于 Summaries 的更新，就是要更新 $item$ 对应的那一项 $summary$ ，这就包括了更新 $count$ 和 $nodes$ 列表。

对于更新 $count$ 值，先判断是否已经为 $item$ 建立了一项，如果建立了直接更新 $count$(即向 $item$ 对应的 $count$ 中增加 add 参数中的 $count$ 值即可)，否则为该元素建立一项并设置 $count$ 值为传入参数的 $count$ 值。

$nodes$ 列表的更新则取决于是否需要新建结点，因为如果是在已有结点上更新，那这个结点一定已经在某个结点链中了，而如果是新建的结点，才需要根据结点的 $item$ 来将新建的结点连入 $item$ 对应的结点链。

更新结点树

首先需要注意的是，事务中的元素是经过排序的，排序是根据元素在整个数据集中出现的频度决定的(其实就是排好序的1项集)，将事务添加到FP树中的过程中，遍历元素的顺序都是按照一个固定顺序进行的，因此对于前缀相同的事务，可以共享前缀中的结点，即只更新结点 count ，只需创建剩余结点。

例如 树上已经有 ${1, 2,5}$这样的路径了，添加新事务${1,2,7}$时，对于1、2，只需要更新结点的 $count$ 值，并增加一个结点7即可，当然，这时候要把结点7连入 item 为7的结点链中。

另外，由于插入的事务在树上一定会形成一个链(其实这也导致了需要使用结点链(这是另一个链…)来将所有 $item$ 相同结点连接起来，因为一棵树上一个 $item$ 有不止一个结点)，因此可以使用一个 $curr$ 来指示插入结点的父结点，这样直接判断 $curr$ 结点是否有 item 子结点即可，上面的例子中，如果已经添加了 1、2 两个结点，那么添加结点 7 时，$curr$ 就是 2 结点，此时 2 的子结点列表中没有 7 结点，因此需要新建一个结点，自然这时候还要将结点 7 插入 $summaries$ 中对应 7 的 $summary.nodes$ 中。

将FP树转换成事务集

FP树中包含了事务集的所有信息，所以也可以从节点树解析出事务集，Spark 中提供了getTransactions方法，这个方法可以求得以某个节点为根节点的子树中的事务集，调用getTransactions(root)即可得到整棵树的事务集。

下面先看下代码，大致了解流程即可：

getTransactions方法是一个递归的方法，它的实现大致如下：假设已经求出一个节点 A 的所有子节点生成的子事务集(可以理解为每个事务都只是一个局部，不完整，下面有例子)，然后将 A 节点连接到子事务集的每个事务前面，这时生成的事务集比刚才的更完整一些，如果A是最后一个需要加入事务的 $item$，那么这时候产生的事务就是完整的事务。

需要注意的是，这个过程中，算法还会一并返回根节点 $count$ 的余量。这个余量是指，生成根节点下方事务之后，根节点还剩余的待生成事务的 $item$ 的数量。例如考虑一个只有一个分支的子树 $(I1:5, I2:1)$ ，调用getTransactions(I1)，算法不但返回事务 $((I2),1)$ ，还会返回一个$(Nil, 4)$ ，因为调用者(上层getTransactions)会在返回值的每一项的事务中加入一个根节点，就是本例中的$I1$。5个$I1$中的一个会和 $((I2),1)$ 生成 $((I1,I2),1)$ ，剩余四个先存成 $(Nil, 4)$ 方便之后使用。

而对于根节点( $null$ )，它的 $count$ 值在之前的 add 函数中提到过，这个值每增加一个事务，就会增加该事务的 $count$ ，结果就是，生成根节点( $null$ )下的事务时，不会有余量，因此不会产生 $(Nil, *)$ 项。

这个函数根据和最小支持度计数生成模式，验证后缀使得函数可以只生成指定后缀的模式，注意事务中的 $item$ 已经排序，所以所有的模式组成的集合可以根据后缀元素不相互覆盖的划分。
